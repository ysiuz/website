Designing an internal comparison tool that helps users put program info in context and decide which they want to save and apply to.

❶ Background

Product Context

Helping families navigate summer education options

BestSummerPrograms is a free web platform built to help students and families navigate the complex & costly landscape of high school summer programs.

Problem

Comparing programs wasn’t natively supported

Comparing program costs, location, affiliation, and other details is essential for informed application decisions, yet we lacked a native way to compare programs.

Existing user flow: Stuttered Comparison

Without a native comparison feature, users had to juggle tabs or take notes outside the platform, creating friction as they browse and choose programs.

Goal

Reduce program comparison friction

Our focus was to support program comparison directly within the platform, making it easier for users to evaluate options and make informed decisions.

Solution

Compare as you browse, accessible across the site

Users can mark programs for comparison from multiple entry points, adding them to a persistent taskbar as they browse. When ready, they can compare selected programs side by side, evaluating them based on their individual needs.

Each colorful node is an Instagram Reel / TikTok that's been submitted to Atlas.

Improved user flow: Fluid Browsing

With a native comparison feature, users can quickly compare programs they’re interested in to see how they stack up and decide which ones are worth applying to.

❷ Desktop Experiences

Checkbox Entry Point

Click Compare Program to get started

I added a Compare checkbox to the action section on program pages, making comparisons accessible sitewide.

Preserving action hierarchy

To keep the primary action (Add to Program List) prominent and avoid clutter, I incorporated Comparison as a secondary action above the action panel.

Before

Sticky panel contains important deadlines and “add to program list button”

After

Sticky panel adds compare checkbox subpanel

Surfacing actions on hover to reduce friction

User testing revealed frustration with needing to open each program page to save or compare programs. We reduced this friction by exposing Save and Compare actions on hover, prior to page entry.

Gallery View Hover: Empowering informed users

Affording users to Save or Compare from the gallery view itself allows users with prior knowledge of programs or universities of interest to act upon it.

Similar Program Hover: Leveraging similarity to prompt action

Because the Similar Programs bar surfaces contextually related programs, extending the hover pattern here allows users to compare high-relevance options without leaving the page

Compare Taskbar

Keeping selected programs in view as you browse

The taskbar provides persistent visual feedback for selected programs, helping users stay oriented as they browse and add find more programs to compare.

Collapsible, yet still informative and interactive

When collapsed, the taskbar shrinks to a minimal strip, offering links to the programs and the ability to remove your selection on hover.

Compare Modal

See program details side by side

The compare modal lays out each program’s core information—badges, affiliation, selectivity, cost, location, and more—so users can compare them in detail and decide which options to pursue.

These users not only share tastes but also submitted content, creating direct links.

Cards as navigational tools

The map and cards were designed to work in tandem. Each interaction within a card re-engages the map, creating a loop between detail and discovery.

Genre Breakdown

Hovering over genres on the user card lights them up on the map.

User Suggestion

Selecting suggested users on your card activates their content cluster on the map.

Surfacing unseen mutuals and genres

Atlas maps what users see and points to what’s missing. By highlighting overlooked genres and distant voices, it nudges towards a fuller view of the algorithmic world.

Suggesting your opposite

Atlas suggests an opposite for each user: someone with very different tastes.

Unexplored Genres

An interactive list of unexplored genres for each user

❸ Results

Takeaway

Prototyping with code is sick :^)

Because of Atlas' niche context and goal, I couldn't rely just on creative software to design my map. I needed to actually code to find the interactions and visual feedback that felt most natural.

Next Steps

Building a more robust onboarding and takeaway

The most common feedback I've received about Atlas is that it's a bit daunting and could use a friendly onboarding. In tandem, it will be great to work on what things users can tangibly takeaway from the experience, like potentially a receipt of their algorithmic profile, or shareable png's.
